# Cafe24 Order-Linked Payment Architecture

> Goal: keep the existing React app and Supabase backend, use Cafe24 as the payment/order channel, and credit points to the app only after server-side payment verification.

---

## 1. Scope

This document defines a production-grade order-linked payment architecture for the current workspace.

Target outcome:

- Existing React app remains the main product UI.
- Supabase remains the system of record for users, app state, and point balances.
- Cafe24 is used for payment and order handling for charge products.
- Point credit happens only after backend verification.
- Duplicate credit, callback loss, and manual reconciliation are handled explicitly.

This architecture replaces the current direct Toss-based payment flow with an order-linked Cafe24 flow.

---

## 2. Current State Summary

Current implementation in this workspace:

- Frontend charge flow starts in `src/app/pages/Points.tsx`
- Payment completion logic is in `src/app/pages/PaymentSuccess.tsx`
- Backend payment endpoints are in `supabase/functions/server/index.tsx`
- Data is mostly stored in KV via `supabase/functions/server/kv_store.tsx`

Current payment model:

1. App creates an order in Supabase Edge Function
2. Frontend opens Toss payment SDK directly
3. Success page calls backend payment confirm
4. Frontend separately calls point add API

Current risks:

- Payment approval and point credit are not handled as a single server-controlled transaction
- Point credit is partially frontend-driven
- Payment provider is hard-wired into the frontend and backend
- KV-only storage is weak for auditability and reconciliation
- Production operations such as retries, reconciliation, and refund handling are not formalized

## 3. Target Principles

The new architecture must satisfy the following.

### 3.1 System of Record

Supabase is the source of truth for:

- internal charge order state
- user point ledger
- payment event history
- reconciliation status

Cafe24 is not the source of truth for point credit. Cafe24 is the external commerce/payment channel.

### 3.2 Order Linking

Every charge attempt must have:

- one internal order ID generated by our backend
- one Cafe24 order ID once checkout is created or payment is completed

These two identifiers must be linked and persisted.

### 3.3 Server-Side Verification

Frontend redirect or success page is not enough to credit points.

Point credit is allowed only after the backend verifies:

- payment status
- amount
- product mapping
- order linkage
- duplicate credit state

### 3.4 Idempotency

The same Cafe24 order must never credit points more than once.

### 3.5 Recoverability

The system must tolerate:

- callback loss
- frontend not returning after payment
- delayed Cafe24 status updates
- temporary external API failure

This requires asynchronous re-check and reconciliation jobs.

---

## 4. High-Level Architecture

```text
React App
  -> Supabase Edge Function: create charge request
  -> Redirect user to Cafe24 checkout/product page
  -> Poll or return to app waiting screen

Cafe24
  -> Creates external order
  -> Accepts payment
  -> Sends completion signal or exposes order/payment status

Supabase Edge Function
  -> Stores internal charge request
  -> Links Cafe24 order to internal order
  -> Verifies payment status through Cafe24
  -> Credits points in ledger
  -> Exposes status to app
  -> Runs reconciliation jobs for missing or delayed events
```

---

## 5. Recommended Flow

### 5.1 Charge Start

1. User opens the points/charge screen in the React app.
2. User selects a charge amount.
3. Frontend calls backend `POST /payments/charges`.
4. Backend validates the amount and maps it to a Cafe24 charge product.
5. Backend creates an internal charge request with status `pending`.
6. Backend returns:
   - `internalOrderId`
   - `checkoutUrl` or redirect payload
   - `expiresAt`
7. Frontend redirects the user to Cafe24 checkout.

### 5.2 Payment Completion

After checkout, one or more of the following may happen:

- Cafe24 sends a completion callback/webhook
- User is redirected to an app return page
- Backend later queries Cafe24 Admin API for order/payment state

The backend stores every incoming or polled payment signal as an event.

### 5.3 Verification and Credit

1. Backend identifies the internal charge request.
2. Backend resolves or confirms the Cafe24 order ID.
3. Backend verifies payment status using Cafe24 order/payment data.
4. Backend verifies:
   - amount matches expected amount
   - product matches expected charge product
   - order is not already credited
5. Backend inserts a point ledger entry.
6. Backend updates charge request status to `credited`.
7. Frontend later reads the final charge status and updates UI.

## 6. Production Flow Variants

### Option A. Redirect + Server Verification

Flow:

- User completes payment in Cafe24
- User returns to app
- App hits backend status endpoint
- Backend verifies Cafe24 order status before credit

Use case:

- acceptable as first production rollout if Cafe24 callback details are limited

Pros:

- simpler implementation
- keeps frontend changes minimal

Cons:

- user must return for immediate completion UX
- requires fallback reconciliation for non-returning users

### Option B. Callback/Webhook + Server Verification

Flow:

- Cafe24 notifies backend after payment completion
- Backend verifies the order via Cafe24 Admin API
- Backend credits points asynchronously
- Frontend only displays status

Use case:

- preferred production mode

Pros:

- backend-controlled completion
- works even if user never returns to app
- better reliability and operations

Cons:

- requires correct callback and signature handling
- requires more integration detail from Cafe24 docs

### Option C. Hybrid

Recommended final production setup:

- use callback/webhook when available
- also allow redirect/return-based status check
- also run scheduled reconciliation

This is the safest model.

## 7. Data Model

The current KV pattern is not enough on its own for payment-grade operations. Production should use structured tables for payment/order state and ledger.

### 7.1 `charge_requests`

Purpose:

- one row per attempted app charge

Suggested fields:

```text
id UUID PK
internal_order_id TEXT UNIQUE NOT NULL
user_kakao_id TEXT NOT NULL
user_id TEXT NULL
amount_krw INTEGER NOT NULL
points INTEGER NOT NULL
product_code TEXT NOT NULL
product_name TEXT NOT NULL
status TEXT NOT NULL
cafe24_order_id TEXT UNIQUE NULL
cafe24_order_status TEXT NULL
cafe24_payment_status TEXT NULL
checkout_url TEXT NULL
verification_attempts INTEGER NOT NULL DEFAULT 0
last_verified_at TIMESTAMPTZ NULL
payment_confirmed_at TIMESTAMPTZ NULL
credited_at TIMESTAMPTZ NULL
failed_reason TEXT NULL
expires_at TIMESTAMPTZ NOT NULL
created_at TIMESTAMPTZ NOT NULL
updated_at TIMESTAMPTZ NOT NULL
```

Allowed statuses:

- `pending`
- `checkout_started`
- `payment_detected`
- `paid`
- `credited`
- `failed`
- `cancelled`
- `expired`
- `refunded`

### 7.2 `payment_events`

Purpose:

- immutable raw event log from all payment-related sources

Suggested fields:

```text
id UUID PK
source TEXT NOT NULL
event_type TEXT NOT NULL
internal_order_id TEXT NULL
cafe24_order_id TEXT NULL
payload JSONB NOT NULL
process_status TEXT NOT NULL
process_message TEXT NULL
received_at TIMESTAMPTZ NOT NULL
processed_at TIMESTAMPTZ NULL
```

Possible `source` values:

- `cafe24_callback`
- `cafe24_webhook`
- `cafe24_redirect`
- `cafe24_polling`
- `admin_manual`

### 7.3 `point_ledger`

Purpose:

- immutable point balance ledger

Suggested fields:

```text
id UUID PK
user_kakao_id TEXT NOT NULL
entry_type TEXT NOT NULL
delta_points INTEGER NOT NULL
internal_order_id TEXT NULL
cafe24_order_id TEXT NULL
description TEXT NOT NULL
created_at TIMESTAMPTZ NOT NULL
```

Recommended uniqueness rule:

- unique on `internal_order_id` where `entry_type = 'charge_credit'`

This is one of the main duplicate-credit guards.

### 7.4 `order_mappings`

Purpose:

- explicit linkage between internal and external orders

Suggested fields:

```text
id UUID PK
internal_order_id TEXT UNIQUE NOT NULL
cafe24_order_id TEXT UNIQUE NULL
mapping_status TEXT NOT NULL
created_at TIMESTAMPTZ NOT NULL
updated_at TIMESTAMPTZ NOT NULL
```

### 7.5 `reconciliation_jobs`

Purpose:

- operational history of automated or manual reconciliation

Suggested fields:

```text
id UUID PK
job_type TEXT NOT NULL
status TEXT NOT NULL
summary JSONB NULL
started_at TIMESTAMPTZ NOT NULL
finished_at TIMESTAMPTZ NULL
```

## 8. Product Mapping Strategy

Each charge amount in the app must map to one Cafe24 charge product.

Example mapping:

```text
charge_10000 -> Cafe24 product A -> 10000 KRW -> 10000 points
charge_30000 -> Cafe24 product B -> 30000 KRW -> 30000 points
charge_50000 -> Cafe24 product C -> 50000 KRW -> 55000 points
```

Do not trust amount coming from the frontend alone.

Backend must own the mapping table:

- input: requested product code or supported amount
- output: exact expected amount, points, and Cafe24 product reference

Recommended location:

- code config first for stability
- database-managed config later if operations need flexibility

---

## 9. Order Linking Strategy

This is the most critical part of the architecture.

### Preferred

Pass `internal_order_id` through the Cafe24 checkout flow and recover it during callback, redirect, or order query.

If Cafe24 allows a merchant reference or custom field, use it.

### Fallback

If direct custom field linking is not available:

- capture Cafe24 order ID at redirect/callback time
- match only against a narrow window of pending internal orders
- validate by user, amount, product, and time window

This fallback should be treated as temporary. Production target should be explicit order linking.

## 10. API Design

### 10.1 `POST /payments/charges`

Purpose:

- create internal charge request

Request:

```json
{
  "amount": 10000,
  "productCode": "charge_10000",
  "kakaoId": "1234567890"
}
```

Response:

```json
{
  "success": true,
  "internalOrderId": "CHG_20260317_ABC123",
  "checkoutUrl": "https://cafe24.example/...",
  "expiresAt": "2026-03-17T15:30:00Z"
}
```

Server behavior:

- validate supported charge amount
- generate internal order ID
- create `charge_requests`
- create `order_mappings`
- return redirect information

### 10.2 `POST /payments/cafe24/callback`

Purpose:

- receive Cafe24 completion signal if available

Server behavior:

- persist raw event in `payment_events`
- extract `internal_order_id` and/or `cafe24_order_id`
- enqueue verification
- return acknowledgement

### 10.3 `POST /payments/cafe24/redirect-complete`

Purpose:

- support user-return flow when frontend comes back from Cafe24

Server behavior:

- record event
- attempt mapping
- trigger verification

### 10.4 `POST /payments/cafe24/verify`

Purpose:

- verify a charge request against Cafe24 order/payment data

Request:

```json
{
  "internalOrderId": "CHG_20260317_ABC123"
}
```

Server behavior:

- load pending charge request
- obtain Cafe24 order/payment state
- compare amount and product
- update status to `paid` on success
- trigger credit step

### 10.5 `POST /payments/charges/:internalOrderId/credit`

Purpose:

- perform point credit transaction

Server behavior:

- ensure request is verified and eligible
- check no existing charge ledger entry exists
- insert `point_ledger`
- update user point balance snapshot if maintained
- mark charge request as `credited`

### 10.6 `GET /payments/charges/:internalOrderId`

Purpose:

- frontend polls current state after redirect or callback

Response:

```json
{
  "success": true,
  "internalOrderId": "CHG_20260317_ABC123",
  "status": "credited",
  "amount": 10000,
  "points": 10000
}
```

## 11. State Machine

```text
pending
  -> checkout_started
  -> expired
  -> cancelled

checkout_started
  -> payment_detected
  -> failed
  -> expired

payment_detected
  -> paid
  -> failed

paid
  -> credited
  -> refunded

credited
  -> refunded
```

Rules:

- only verified payments can move to `paid`
- only `paid` can move to `credited`
- `credited` is terminal for normal success path

---

## 12. Verification Rules

Before point credit, backend must verify all of the following.

- internal order exists
- internal order is not expired or cancelled
- internal order is not already credited
- Cafe24 order exists
- Cafe24 payment status is paid/confirmed
- Cafe24 amount equals expected amount
- Cafe24 product/order item matches expected charge product
- linked user identity is acceptable according to chosen mapping strategy

If any verification fails:

- store failure reason
- keep event log
- do not credit points

---

## 13. Idempotency Rules

The backend must enforce multiple duplicate prevention layers.

### Layer 1. Charge Request Status Guard

Do not process credit unless current charge status is eligible.

### Layer 2. Ledger Uniqueness Guard

Unique ledger row per internal charge order.

### Layer 3. Cafe24 Order Uniqueness Guard

Each Cafe24 order ID must map to only one internal order.

### Layer 4. Event Replay Tolerance

Callbacks and verification jobs must be safe if repeated.

---

## 14. Reconciliation

Production must not rely on callback success alone.

Recommended scheduled jobs:

### 14.1 Pending Recheck

Every 1 to 5 minutes:

- find `checkout_started` or `payment_detected`
- re-query Cafe24 order/payment state
- complete verification if possible

### 14.2 Paid But Not Credited Recheck

Every 1 minute:

- find charges with `paid` but no `credited_at`
- retry credit transaction

### 14.3 Daily Reconciliation

At least once per day:

- compare internal paid/credited state with Cafe24 order/payment history
- identify:
  - paid in Cafe24 but missing internal credit
  - credited internally but refunded in Cafe24
  - unmatched Cafe24 orders

Store reconciliation summary in `reconciliation_jobs`.

---

## 15. Refund and Cancellation Handling

Refund support must exist before production launch.

Required behavior:

- detect Cafe24 refund/cancel state
- update charge request to `refunded` or `cancelled`
- if points were already credited:
  - create compensating ledger entry such as `refund_debit`
  - preserve audit history

Policy decisions required:

- what if user has already spent some credited points
- whether refund creates negative point balance
- whether such cases go to manual review queue

## 16. Frontend Responsibilities

Frontend should become thinner than the current Toss flow.

Frontend should:

- present supported charge options
- call charge creation API
- redirect to Cafe24 checkout
- display waiting/processing UI on return
- poll charge status until final state

Frontend should not:

- perform payment confirmation logic
- directly add points after return
- decide final payment success by query string alone

### Required changes from current code

#### `src/app/pages/Points.tsx`

Replace:

- Toss SDK loading
- Toss `requestPayment()`

With:

- backend charge creation call
- redirect to Cafe24 checkout URL

#### `src/app/pages/PaymentSuccess.tsx`

Replace:

- provider-specific confirm call
- frontend-driven `addPoints()`

With:

- status polling page for internal order
- final success/failure display only

## 17. Backend Responsibilities

Backend must own all sensitive logic.

Backend must:

- create internal orders
- validate supported charge products
- map internal order to Cafe24 order
- receive callbacks and redirects
- verify payment state against Cafe24
- write point ledger
- expose charge status
- run reconciliation

### Required changes from current code

#### `supabase/functions/server/index.tsx`

Current Toss-specific endpoints:

- `/payment/create-order`
- `/payment/confirm`

Target replacement:

- `/payments/charges`
- `/payments/cafe24/callback`
- `/payments/cafe24/redirect-complete`
- `/payments/cafe24/verify`
- `/payments/charges/:internalOrderId`

#### `supabase/functions/server/kv_store.tsx`

Current KV helper may still be used for transitional data, but payment-grade records should move to structured tables for:

- charge requests
- payment events
- point ledger
- reconciliation jobs

## 18. Security Requirements

### 18.1 Secret Management

Do not hardcode:

- admin secret
- provider secret keys
- callback verification keys

Use environment secrets only.

### 18.2 Callback Validation

If Cafe24 provides signature validation or trusted headers:

- verify them
- reject unverifiable callbacks

### 18.3 Least Privilege

Separate:

- public frontend token usage
- privileged server operations
- admin operations

### 18.4 Auditability

Persist raw event payloads and all state transitions.

## 23. Bottom Line

The correct production target is not "price click -> direct URL mapping -> trust redirect".

The correct production target is:

1. create internal charge order
2. send user to Cafe24 checkout
3. link Cafe24 order to internal order
4. verify payment server-side
5. credit points through immutable ledger
6. reconcile missing or delayed events automatically

That model preserves the existing app while making Cafe24 a payment/order channel rather than the core application backend.
